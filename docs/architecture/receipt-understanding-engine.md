# Receipt Understanding Engine

## Status and decisions

Phase 8.0 implements RFC-005 and the approved architecture decisions:

- ADR-030: receipt understanding classifies document structure, never business entities.
- ADR-031: every OCR block maps to exactly one receipt element; block splitting and merging belong to the future Receipt Line Builder.

The engine answers only: **What kind of receipt element is this?**

## Dependency flow

```text
Receipt image
  -> Receipt OCR platform
  -> ReceiptUnderstandingService
  -> ReceiptUnderstandingEngine
  -> immutable ReceiptUnderstandingResult
```

The domain and engine depend only on provider-independent OCR domain models. They do not import Flutter, ML Kit, repositories, inventory, products, matching, purchases, shopping, or Pantry Intelligence services. Results are derived in memory and are never persisted.

## Domain result

One `ReceiptOcrBlock` always produces one immutable `ReceiptElement`. Each element preserves:

- a deterministic identifier;
- original recognized text;
- nullable provider-independent bounding region;
- nullable OCR confidence;
- exactly one structural type;
- immutable `ReceiptElementEvidence` generated by the engine.

Supported types are exactly `Unknown`, `StoreName`, `Header`, `ProductName`, `Price`, `Quantity`, `Discount`, `Tax`, `Total`, `Metadata`, and `Footer`.

Evidence records normalized text, relative position, matched structural patterns, neighbouring element references, OCR confidence, and an optional engine summary. Confidence is evidence only and never changes classification.

## Ordering and layout

When OCR guarantees reading order, that order is preserved. Otherwise, the service applies one canonical O(n log n) spatial sort:

1. top to bottom;
2. left to right for Latin text or right to left for Arabic text;
3. original OCR index as the deterministic tie-breaker;
4. blocks without usable geometry after bounded blocks, retaining stable input order.

The document bounds define three positional zones by block centre:

- top 20 percent: header zone;
- middle 60 percent: body zone;
- bottom 20 percent: footer zone.

Missing or invalid geometry is valid input and produces `relativePosition = Unknown`.

## Classification

The engine normalizes English casing, Arabic character variants, Arabic and Persian digits, punctuation, and whitespace for structural comparison while retaining the original OCR text in the result. Only generic Arabic and English receipt-structure terms are allowed; retailer names, product names, and other business dictionaries are forbidden.

Rules execute with fixed precedence:

1. total;
2. tax;
3. discount;
4. quantity;
5. price;
6. metadata;
7. structurally prominent top-zone store name;
8. header;
9. footer;
10. structurally product-like body text;
11. unknown.

`StoreName` is only the one structurally prominent top-zone text. `ProductName` means only product-like body text near monetary structure; it does not identify or match a catalog product.

## Deterministic identity

Element IDs use an unsigned 64-bit FNV-1a hash of normalized text and normalized bounding-box data. Repeated identical keys receive deterministic occurrence suffixes. Identical ordered inputs therefore produce identical classifications, evidence, and identifiers.

## Service and failure boundary

`ReceiptUnderstandingService` accepts a completed `ReceiptOcrResult`, orders its blocks once, invokes the engine once, and verifies the one-to-one output invariant. Unexpected classification exceptions and invariant violations are converted to provider-independent `ReceiptUnderstandingFailure` values.

OCR provider and image failures are mapped by the upstream Receipt OCR platform before this service is invoked; no SDK exception or SDK type crosses this boundary.

## Presentation

The read-only debug screen exposes loading, results, empty, and error states. It can show original OCR blocks, classified elements, type filters, confidence, engine evidence, and bounding-box overlays. It contains no classification rules and provides no editing or business actions.

The existing successful OCR screen offers an optional route to this debug view. The established product-matching and receipt-import flow remains unchanged.

## Complexity and integrity

- Classification is one pass over the ordered blocks: O(n).
- Canonical ordering, when required, is O(n log n).
- No repository access, persistence, cache, mutation, repeated document scan, or Flutter dependency exists inside the engine.
- Original OCR collections and values are not mutated.

## Intentionally deferred

Receipt Line Builder block splitting/merging, receipt item construction, product matching, purchase creation, inventory changes, receipt validation, AI/LLM classification, cloud processing, and business-entity detection are outside Phase 8.0.
